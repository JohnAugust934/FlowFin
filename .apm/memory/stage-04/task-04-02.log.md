# Task 4.2 — Metas com Propósito, Simulador, Prioridades & Investimentos

## Summary
Implementado o Pilar 5: CRUD de metas nomeadas com propósito, prazo e prioridade
(com progresso e ordenação por prioridade), simulador determinístico de metas e CRUD
de investimentos com total agregado. Tudo escopado ao usuário, soft delete, valores em
centavos. Suíte completa verde (145/145).

## Details
- **Metas (`GoalController` + `GoalResource`):** CRUD escopado a `user_id`, soft delete.
  Reaproveita o schema existente de `goals` (Task 1.3: `name`, `description`, `target_amount`,
  `saved_amount`, `due_date`, `priority` enum baixa/media/alta). **Nenhuma migration aditiva
  foi necessária** — `description` carrega o "propósito" e `due_date`/`priority` já existem.
  - **Progresso:** `progress_pct = min(100, saved/target·100)`, mais `remaining_amount`, no Resource.
  - **Ordenação por prioridade:** `alta → media → baixa` (CASE), depois prazo mais próximo
    (metas com prazo antes das sem prazo), depois id desc. Listagem paginada 20/página.
  - FormRequests `StoreGoalRequest`/`UpdateGoalRequest` (update parcial via `sometimes`),
    com coerção de R$→centavos para `target_amount`/`saved_amount`.
- **Simulador (`SimulatorService` + `SimulatorController`):** dados DOIS dos três valores,
  calcula o terceiro. **Fórmulas exatas (poupança simples, sem juros):**
  - meses = `ceil(alvo / mensal)`; alvo = `mensal × meses`; mensal = `ceil(alvo / meses)`.
  - Arredondamento **para cima** (ceil) nos casos com divisão: garante que o alvo é de fato
    alcançado (nunca subestima). `POST /api/goals/simulate` (rota estática antes de `/{id}`),
    `SimulateGoalRequest` exige exatamente dois campos preenchidos. Resposta inclui `computed`
    (qual campo foi calculado).
- **Investimentos (`InvestmentController` + `InvestmentResource` + `InvestmentFactory`):** CRUD
  escopado, soft delete. Reaproveita o schema de `investments` (Task 1.3: `description`, `type`,
  `amount`). Listagem paginada 20/página com **`total_invested`** (soma de TODOS os investimentos
  do usuário, não só da página) exposto como chave de topo da resposta — não sobrescreve o `meta`
  de paginação. FormRequests com coerção de R$→centavos para `amount`.

## Important Findings
- **Contratos JSON para a UI 4.4** (valores em CENTAVOS; respostas de Resource encapsuladas em
  `data` — wrapping padrão do projeto está ATIVO):
  - `GET /api/goals?page=` → `{ data:[ { id, name, description, target_amount, saved_amount,
      progress_pct, remaining_amount, due_date(aaaa-mm-dd|null), priority, created_at, updated_at } ], links, meta }`
      (ordenado por prioridade, depois prazo).
  - `POST /api/goals` → 201 `{ data:{...} }`; corpo `{ name, description?, target_amount,
      saved_amount?, due_date?, priority?('baixa'|'media'|'alta', default 'media') }`.
  - `GET|PUT|DELETE /api/goals/{id}` → detalhe / update parcial / soft delete (`{message}`).
  - `POST /api/goals/simulate` → `{ monthly_amount, target_amount, months, computed }`;
      corpo com **exatamente dois** de `{monthly_amount, target_amount, months}`.
  - `GET /api/investments?page=` → `{ data:[ {id,description,type,amount,...} ], links, meta, total_invested }`.
  - `POST /api/investments` → 201 `{ data:{...} }`; `PUT|DELETE /api/investments/{id}`.
- **Divergência da Spec a confirmar com o Manager:** a Spec cita "propósito" como atributo da
  meta. Mapeei o **propósito no campo `description`** (já existente) em vez de criar uma coluna
  nova, evitando migration redundante. Se o produto quiser um campo dedicado `purpose` separado
  da descrição, é migration aditiva simples.
- **Decisão do Worker:** simulador arredonda meses e valor mensal **para cima** (ceil) para nunca
  subestimar o esforço necessário a atingir o alvo. Se a Spec preferir arredondamento "natural"
  (round) ou exibir fração, é ajuste pontual.
- **Prioridade** é o enum `baixa|media|alta` já existente; a ordenação usa CASE explícito
  (compatível com SQLite dos testes e MySQL da hospedagem).
