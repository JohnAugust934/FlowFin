# Task 4.1 — Score, Streak, Dicas & Conteúdo Educativo

## Summary
Implementada a gamificação e o direcionamento do Pilar 4 (Mentalidade), tudo
**determinístico** (sem ML): Score FlowFin mensal (0–100), sequência ("streak") de
dias com registro e seu reset via scheduler, motor de dicas contextuais e
mini-conteúdos educativos (seed). Endpoints JSON escopados ao usuário. Suíte completa
verde (145/145).

## Details
- **Score (`App\Services\ScoreService`):** média ponderada de 3 fatores normalizados 0–100:
  - **Consistência (peso 40%, sempre presente):** dias do mês com ≥1 registro ÷ dias do mês
    (denominador = dias de calendário do mês — decisão do Worker, mantém a métrica estável
    e comparável entre meses).
  - **Orçamentos respeitados (peso 30%):** orçamentos com `consumido ≤ limite` ÷ total de
    orçamentos. **Excluído** (renormaliza) quando não há orçamento. Reaproveita `BudgetService`.
  - **Progresso na meta (peso 30%):** `progress_pct` capado a 100. **Excluído** quando não há
    meta. Reaproveita `SavingsGoalService`.
  - **Renormalização:** `score = Σ(valor·peso incluídos) / Σ(peso incluídos)`, arredondado.
    Cada fator expõe `{value, weight, included}`. Não tem cache próprio — compõe agregados já
    cacheados (Dashboard/Budget/SavingsGoal), invalidando-se de forma coerente com eles.
- **Streak (`App\Services\StreakService`):** dias consecutivos com ≥1 registro. O dia corrente
  conta como "em aberto": enquanto não houver registro hoje, a sequência até ontem permanece
  válida; quebra quando um dia inteiro passa sem registro. `compute()` é a fonte de verdade
  (recalcula das transações); `refresh()` persiste o snapshot em `users.current_streak`.
- **Job/scheduler (`App\Jobs\RecalculateStreaks`, `ShouldQueue`):** recalcula/persiste o streak
  de todos os usuários em lotes (`chunkById`). Registrado em `routes/console.php` via
  `Schedule::job(...)->dailyAt('00:10')` — confirmado em `php artisan schedule:list`. É o que
  **efetiva o reset** diário. Despacho via **fila no banco** (QUEUE_CONNECTION=database em prod;
  cron único dispara `schedule:run`). Nos testes a fila é `sync`.
- **Dicas (`App\Services\TipsService`):** lista priorizada (alertas → positivos → educativos),
  PT-BR, em linguagem humana. Alertas: orçamento estourado, gasto acima do mês anterior.
  Positivos: streak ativo (≥2), meta atingida. Educativos sempre presentes: reserva de
  emergência e regra 50-30-20. Compõe Insights/Budget/SavingsGoal/Streak sem recalcular nada.
- **Conteúdo educativo:** `EducationalContentSeeder` (5 mini-conteúdos idempotentes, 1 parágrafo
  cada — reserva de emergência, 50-30-20, hábito, gastos invisíveis, metas com propósito),
  registrado no `DatabaseSeeder`. Endpoint de listagem paginado (20/página) com filtro por tema.

## Important Findings
- **Contratos JSON para a UI 4.3** (valores em CENTAVOS; `month=aaaa-mm`):
  - `GET /api/score?month=` → `{ month, score (0–100 int), factors: {
      consistency:{value,weight,included}, budgets:{value|null,weight,included},
      savings_goal:{value|null,weight,included} } }`. `included=false` ⇒ fator excluído e pesos
      reescalados.
  - `GET /api/streak` → `{ current_streak, last_activity_date (aaaa-mm-dd|null), active }`.
  - `GET /api/tips?month=` → `[ { code, level('alerta'|'positivo'|'educativo'), title, message, theme } ]`.
  - `GET /api/educational-contents?theme=&page=` → coleção paginada `{ data:[{id,title,theme,body}], links, meta }`.
- **Decisões do Worker a confirmar com o Manager:**
  1. **Denominador da consistência = dias de calendário do mês** (não "dias decorridos até hoje").
     No mês corrente, isso faz o fator começar baixo e subir ao longo do mês. Se a Spec preferir
     "dias decorridos", é troca de uma linha.
  2. **Meta ausente → fator excluído (renormaliza)**, análogo a "sem orçamento". A consistência
     está sempre presente, então o Score nunca fica indefinido.
  3. **Limiar do streak para dica positiva = 2 dias** (evita parabenizar 1 dia isolado).
- **Reset do streak:** materializado pelo job diário (snapshot em `users.current_streak`); o
  endpoint devolve sempre o valor recalculado ao vivo. Prod precisa do **cron único** chamando
  `php artisan schedule:run` e de `QUEUE_CONNECTION=database` + tabela `jobs` (já existe) — a
  ativação do cron é tarefa de deploy (orientar o usuário).
