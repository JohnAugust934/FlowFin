# Task 3.2 - Relatório "Onde Economizar"

## Summary
Entregue o serviço + endpoint JSON do relatório "Onde economizar" (Pilar 3 — Economia): sugestões **determinísticas** (sem ML) de onde cortar gastos, geradas a partir das maiores categorias de saída classificadas como "Desejo" e dos gastos recorrentes ativos, cada uma com corte percentual, economia mensal estimada (centavos) e mensagem em linguagem humana, priorizadas pela maior economia potencial. Suíte completa 100% verde.

## Details
- **Serviço (`App\Services\SavingsReportService`):** `forUser(User, ?month)` devolve, num payload cacheado (`savings_report:{user}:{month}`), `total_potential_savings`, `count` e `suggestions[]` ordenadas por `estimated_savings` desc. Duas fontes compostas sobre dados já existentes (sem reimplementar):
  - **Categorias de "Desejo":** soma de saídas com `classification='desejo'` agrupadas por categoria no mês → corte **30%** → economia estimada `round(total*0.30)`.
  - **Gastos recorrentes:** recorrências de saída ativas, normalizadas para impacto mensal (mesma fórmula da 3.1: diária×365/12, semanal×52/12, mensal×1, anual÷12) → corte **20%** → economia estimada.
- **Endpoint:** `GET /api/savings-report?month=aaaa-mm` (default = mês atual), autenticado por sessão, escopado a `user_id`, no grupo `api` de `routes/web.php`. Controller `Api\SavingsReportController` valida `month` (`date_format:Y-m`).
- **Linguagem humana (PT-BR):** campo `message` pronto para exibição — categorias: "Reduza 30% em {categoria} e economize R$ X por mês."; recorrentes: "Revise {descrição} (gasto recorrente): cortar 20% economiza R$ X por mês." Valor formatado via `App\Support\Money::formatBRL`.
- **Cache/invalidação:** `TransactionObserver` estendido para invalidar também `savings_report` do mês afetado (a parte "desejo" depende de transações). `RecurrenceObserver` estendido para invalidar o relatório do mês corrente (a parte "recorrente" depende das recorrências) além dos invisíveis.

## Important Findings
- **Percentuais de corte são decisão do Worker (a Spec só dá 30% como exemplo):** categorias de "Desejo" = **30%** (alinhado ao exemplo "Reduza 30% em delivery"); gastos recorrentes = **20%** (compromissos fixos são mais difíceis de cortar; representa renegociação/troca de plano em vez de cancelamento total). Se o produto quiser outros percentuais (ou percentuais variáveis por faixa de valor), é ajuste de constante — confirmar com o Manager.
- **Contrato JSON para a UI 3.4 (Economia)** — valores em CENTAVOS inteiros, `month=aaaa-mm`:
  - `GET /api/savings-report?month=aaaa-mm` → `{ month, total_potential_savings, count, suggestions: [ { type:('categoria_desejo'|'recorrente'), reference_id, label, current_amount, cut_pct, estimated_savings, category:{id,name,icon,color}|null, message } ] }`
  - `reference_id` é o id da **categoria** (quando `categoria_desejo`) ou da **recorrência** (quando `recorrente`); `current_amount` é o gasto do mês na categoria ou o impacto mensal da recorrência.
- **Possível sobreposição categoria×recorrente:** uma despesa recorrente pode também gerar transações de saída classificadas como "desejo". Como recorrências e transações são entidades separadas, uma mesma despesa pode aparecer nas duas listas (uma vez como categoria de desejo, outra como recorrente). Mantido separado por ser determinístico e transparente; se o produto quiser deduplicar, é regra adicional — sinalizado para o Manager.
- **Limitação de invalidação do cache do relatório:** o relatório é cacheado por mês, mas a contribuição das recorrências é a mesma em todos os meses. O `RecurrenceObserver` invalida apenas o mês corrente ao alterar uma recorrência; relatórios de meses passados recalculam na próxima alteração de transação daquele mês. Mesmo trade-off já documentado para orçamentos na 3.1; aceitável para o uso real.
- **Escopo:** o relatório consome a classificação necessidade/desejo (Tasks 2.1/2.2) e as recorrências (3.1) — nada foi reimplementado. A UI 3.4 consumirá este endpoint junto com orçamentos/meta/invisíveis da 3.1.
