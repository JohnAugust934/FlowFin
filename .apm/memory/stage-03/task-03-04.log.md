# Task 3.4 - UI Economia

## Summary
Tela **Economia** (`/economia`) construída consumindo `GET /api/savings-goal`, `GET /api/budgets/status` (+ CRUD `/api/budgets`), `GET /api/insights/invisible` e `GET /api/savings-report`. Quatro seções no idioma visual do dashboard: **meta de economia** (barra de progresso + estado "atingida"), **orçamentos por categoria** (badges/barras semafóricos + definir/editar/excluir limite), **gastos invisíveis** (recorrências com impacto mensal somado) e **onde economizar** (sugestões prontas em PT-BR com economia destacada). Build compila, 107 testes verdes. Resta validação visual guiada (status Partial).

## Details
- **Rota/navegação:** `Route::view('/economia', 'savings.economia')->name('savings.economia')` no grupo `auth`. Ligada no bottom-nav mobile e na barra desktop (ver log 3.3 para a reorganização da navegação).
- **API client** (`api.js`): `getBudgetsStatus(month)`, `createBudget`, `updateBudget(id)`, `deleteBudget(id)`, `getSavingsGoal(month)`, `updateSavingsGoal(cents)` (corpo `{ monthly_savings_goal }`, `null` limpa), `getSavingsReport(month)`. Valores em centavos; conversão R$↔centavos via `brlToCents`/`maskCurrency`/`centsToBRL` existentes.
- **Componente Alpine `economia`** (`components.js`): carrega tudo em paralelo (`Promise.all`) por mês; recarrega ao ouvir `transaction-saved`; navegação de mês como o dashboard.
  - **Meta:** `hasGoal`, `goalView` (rótulos, `width` capado 0–100, `achieved`, faltante). Form com máscara R$ → `updateSavingsGoal`; botão "Remover meta" envia `null`. Aviso quando `saved` é negativo (saídas > entradas, sem sobra).
  - **Orçamentos:** `budgetRows` (cor/ícone, `width` capado, `pctLabel`, `overspent`, classes de barra/badge/label por `status ∈ {ok,alerta,estourado}`). `availableCategories` = categorias sem orçamento (para o seletor de novo). Form único cria/edita (ao editar, categoria fixa exibida desabilitada; limite via PUT). Exclusão com confirmação inline. Erros 422 mapeados por campo (`budgetError`).
  - **Invisíveis:** `invisibleItems` (impacto mensal formatado + `frequencyLabel` em PT-BR), `invisibleTotalLabel`. Nota explicando que valores são normalizados para o equivalente por mês.
  - **Onde economizar:** `suggestions` (usa `message` pronta do backend), `potentialSavingsLabel` destacado.
- **View** (`resources/views/savings/economia.blade.php`): pílula de mês, loading/erro globais, `x-cloak`; cada seção com estado vazio próprio. Barras/badges semafóricos seguem os tokens do design system (verde `bg-gradient-emerald`/<80%, amarelo `bg-warning`/80–99%, vermelho `bg-danger`/≥100%). Feedback de escrita por toast (`success`/`error`) em meta e orçamentos.

## Important Findings
- **Decisão de UX (registrar):** os **gastos invisíveis** ficaram nesta tela (3.4), não na Consciência (3.3) — par temático com "onde economizar".
- **Não há tela de cadastro de recorrências** ligada na navegação; o painel de invisíveis depende de transações marcadas como recorrentes (`is_recurring` no formulário rápido). Se ficar vazio na validação, é esperado até o usuário marcar uma conta fixa.
- **Reusos do design system:** badges/barras semafóricos replicam os tokens dos componentes `badge`/`progress` inline no `x-for` (Blade components não recebem estado Alpine dinâmico). Comportamento idêntico ao design system.
