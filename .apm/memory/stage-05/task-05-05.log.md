# Task 5.5 — Hardening de Performance

## Summary
Auditadas as consultas-chave que alimentam as telas (dashboard, insights, relatório
"onde economizar", score/metas/investimentos). O código **já estava livre de N+1**
(eager loading consistente) e com paginação 20/página. O ganho real foi um **índice
composto** em `transactions` para o padrão de filtro dominante, com teste anti-N+1 e
medição de tempo. Suíte completa verde (155/155).

## Details
- **Auditoria N+1 (sem alterações necessárias):**
  - `DashboardService` — `by_category` faz 1 consulta de totais + 1 `whereIn` para as
    categorias (sem laço por linha). Somas agregadas no banco.
  - `InsightsService` (top 3, recorrentes), `SavingsReportService` (categorias desejo,
    recorrentes) — todos usam `with('category')` antes do `->map()`. Sem N+1.
  - `MonthlyReportService` (5.3) — `with('category:id,name')`.
  - Confirmado por **teste de contagem de queries**: dashboard com 8 categorias × 120
    transações executa ≤ 10 consultas (constante, não cresce com o volume).
- **Índice composto (migration aditiva):** as consultas filtram sempre
  `user_id` (igualdade) + `type` (igualdade) + `date` (faixa do mês). Os índices de coluna
  única existentes deixam o MySQL usar só **um** por consulta. Adicionado
  `transactions_user_type_date_index (user_id, type, date)` — colunas de igualdade antes da
  de faixa. **Aditivo:** não remove os índices existentes.
- **Paginação:** confirmada em 20/página nas listagens (transações, metas, investimentos,
  conteúdos, orçamentos) — já vigente das Stages anteriores.
- **Cache de agregados:** preservado — dashboard via `Cache::rememberForever` por usuário/mês,
  invalidado pelos Observers (`#[ObservedBy]`). Não alterado.

## Validation
- `php artisan test` → **155/155 passando, 498 asserções**.
- **Medição no MySQL** (benchmark via tinker):
  - 1 usuário, **5.000 transações** num mês: dashboard **cold compute ≈ 26 ms**, **warm cache
    ≈ 0,47 ms** — ambos muito abaixo do orçamento de **2–3 s** no celular.
  - `EXPLAIN` do filtro principal lista o novo índice em `possible_keys`
    (`transactions_user_type_date_index`); com `user_id` seletivo (multiusuário) o planner
    restringe a varredura às linhas do usuário (rows examinados caem ao conjunto do usuário).
- Sem N+1 nas queries-chave (verificado por log de queries no teste automatizado).

## Issues
None.

## Important Findings
- **N+1 já eliminado** em toda a base auditada — nenhuma correção de eager loading foi
  necessária; o índice composto é a melhoria incremental que importa em escala (muitos
  usuários × muitas transações por mês), quando a faixa de `date` passa a discriminar.
- **Decisão do Worker:** ordem do índice `(user_id, type, date)` — igualdade antes de faixa —
  é o arranjo ótimo para o padrão de consulta; mantive os índices de coluna única para não
  regredir outras queries (aditivo).
- **Fora de escopo (confirmado na Spec):** minificação Vite e Gzip são da Task 6.1, não tratadas
  aqui.
- **Para deploy:** rodar `php artisan migrate` aplica o índice (migration aditiva, sem perda de
  dados).
