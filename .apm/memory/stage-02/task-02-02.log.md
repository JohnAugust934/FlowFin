# Task 2.2 - Serviço de Agregados do Dashboard

## Summary
Serviço de agregados mensais (entrou/saiu/sobrou, por categoria, % necessidade/desejo) com cache em arquivo por usuário+mês, invalidado a cada mutação de transação via observer, exposto por endpoint JSON autenticado. 61/61 testes passando.

## Details
Worktree `C:\_PROJETOS\FlowFin\.apm\worktrees\dashboard-agregados`, branch `feature/dashboard-agregados`. Bootstrap: `.env` copiado, `composer install`, `npm install`+`npm run build` (necessário para os testes de página do Breeze que usam `@vite`). Construído sobre o model `Transaction` e os padrões da API da Task 2.1.

- **`app/Services/DashboardService.php`:** `forUser(User, ?month)` normaliza o mês ("aaaa-mm", default = atual) e serve via `Cache::rememberForever`. Cálculos por **somas agregadas no banco** (sem laço PHP): entrou/saiu via `SUM(amount)` por `type`; `by_category` via `GROUP BY category_id` + uma única consulta para carregar nome/ícone/cor das categorias (evita N+1), ordenado por total desc, com `category_id` nulo rotulado "Sem categoria"; `needs_vs_wants` via `GROUP BY classification` sobre saídas classificadas. Refatorei os helpers para receber a **factory de query** (`callable $base`) em vez de um Builder compartilhado — evitando um bug onde clonar um builder já filtrado por `whereNotNull('classification')` e adicionar `whereNull('classification')` zeraria `sem_classificacao`.
- **Tratamento documentado:** saídas sem `classification` são **excluídas** do cálculo de % (mas contam em "saiu" e no resumo por categoria); o total delas é exposto em `needs_vs_wants.sem_classificacao`.
- **`app/Observers/TransactionObserver.php`:** invalida o cache em `created/updated/deleted/restored/forceDeleted`. Em `updated`, se a `date` mudou de mês (`wasChanged('date')`), invalida também o mês de origem (`getOriginal('date')`). Registrado via atributo `#[ObservedBy([TransactionObserver::class])]` no model `Transaction` — cobre todos os caminhos de mutação (API da 2.1, futuras telas, sync offline) de forma robusta.
- **`app/Http/Controllers/Api/DashboardController.php`:** `GET /api/dashboard?month=aaaa-mm` (default = mês atual), validação `date_format:Y-m`, escopado ao usuário autenticado. Rota adicionada ao grupo `auth`/`api` (`api.dashboard`).
- Valores monetários permanecem **centavos inteiros** em todo o payload.

## Contrato do Endpoint (para a UI do dashboard — Task 2.4)
`GET /api/dashboard?month=aaaa-mm` (sessão autenticada, `Accept: application/json`; default = mês atual). Resposta (valores em **centavos inteiros**):
```
{
  "month": "2026-06",
  "totals": { "entrou": int, "saiu": int, "sobrou": int },
  "by_category": [ { "category_id": int|null, "name": str, "icon": str|null, "color": str|null, "total": int } ],
  "needs_vs_wants": { "necessidade": int, "desejo": int, "sem_classificacao": int,
                      "necessidade_pct": float, "desejo_pct": float }
}
```
`month` inválido → 422 (`date_format:Y-m`). Sem auth → 401. `by_category` ordenado por `total` desc.

- **Invalidação do cache** verificada: `Cache::has(key)` verdadeiro após primeiro cálculo; falso após criar/editar/excluir transação; recálculo reflete o novo valor. Edição que move a transação entre meses invalida **ambos** os meses.
- Endpoint: contrato/estrutura, default de mês atual, 422 para mês inválido, escopo por usuário (totais do outro usuário não aparecem), 401 sem auth.

## Important Findings
- **Contrato do endpoint do dashboard documentado acima** — é a dependência da UI do dashboard (Task 2.4). Valores em centavos; formatação R$ é da UI.
- **Observer registrado no model `Transaction` via `#[ObservedBy]`** (artefato originado na Task 1.3, evoluído na 2.1). A invalidação é independente dos controllers, então qualquer caminho de mutação (incluindo a futura sincronização offline) mantém os agregados consistentes. Workers que mexerem no model devem preservar esse atributo.
- **Driver de cache:** usa o cache padrão do Laravel (em produção, `file`, conforme restrição de hospedagem; em testes, `array`). Nenhuma dependência de Redis introduzida.
- **Build de assets (`npm run build`) é pré-requisito da suíte completa** (testes de página do Breeze usam `@vite`) — mesmo ponto da Task 1.4; relevante para CI/deploy.
