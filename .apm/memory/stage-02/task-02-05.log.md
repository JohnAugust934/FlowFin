# Task 2.5 - Histórico com Filtros

## Summary
Entregue a tela de histórico de transações (`/transacoes`) com lista paginada (20/página), filtros por período/categoria/tipo, e ações de editar (reaproveitando o formulário da 2.3) e excluir (soft) com confirmação e feedback. **Achado importante:** a API `GET /api/transactions` da 2.1 ainda não aceita parâmetros de filtro no servidor — os filtros são enviados como query params (prontos para o backend) e aplicados como fallback no cliente sobre a página carregada; recomenda-se um ajuste de backend.

## Details
Mesma branch/worktree do lote (`feature/ui-registro-historico`). Tela em `resources/views/transactions/history.blade.php`, componente Alpine `transactionHistory` (em `resources/js/flowfin/components.js`).

- **Lista:** consome `GET /api/transactions`; cada item mostra ícone/cor da categoria, descrição (ou nome da categoria), categoria, data em **dd/mm/aaaa**, classificação (quando houver) e valor formatado em **R$** com sinal/cor (+ verde para entrada, − vermelho para saída). Estados de loading e vazio (com CTA para registrar a primeira).
- **Paginação:** usa `meta` (`current_page`, `last_page`, `per_page`, `total`) com botões Anterior/Próxima e indicador "Página X de Y".
- **Editar/Excluir:** editar dispara `edit-transaction` (abre o formulário global da 2.3 em modo edição → `PUT`); excluir tem confirmação inline e usa `DELETE` (soft), com toast e recarga da lista (volta uma página se a atual ficar vazia). A lista também escuta `transaction-saved` para refletir criações/edições feitas em qualquer lugar.
- **Consistência:** reusa o design system (cards, botões, badges, cores) e linguagem humana ("entradas"/"saídas").

### Filtros — desvio do contrato (relevante)
Li o `TransactionController@index` (2.1): ele apenas pagina (`latest('date')->paginate(20)`), **sem ler nenhum query param**. Conforme a instrução da Task ("priorize o que a API suporta e implemente o filtro no cliente apenas como fallback bem documentado"), a implementação:
1. Envia os filtros como query params no `GET /api/transactions` (`date_from`, `date_to`, `category_id`, `type`) — encaixe imediato quando o backend passar a suportá-los.
2. Aplica os mesmos filtros no cliente sobre a página recebida (`applyClientFilters`) como fallback.

Limitação: como o servidor não filtra, o fallback atua apenas sobre os 20 itens da página atual e a `meta` reflete o total não filtrado. Para bases com >20 itens, o filtro fica correto apenas por página. **Recomendação de backend** (ajuste posterior): aceitar `date_from`, `date_to`, `category_id`, `type` em `TransactionController@index`, mantendo a paginação de 20.

## Compatibility Concerns
- A navegação compartilhada (`bottom-nav`, `navigation`) passou a apontar "Transações" para `/transacoes`.
- Editar depende do formulário global da 2.3 (`x-transaction-form` no app shell) e do evento `edit-transaction`.

## Important Findings
- **A API de listagem não suporta filtros server-side** (sugere ajuste de backend: `date_from`, `date_to`, `category_id`, `type` no `index`, com paginação). Até lá, o filtro do histórico é page-local (fallback documentado). Isso afeta o Backend Agent / coordenação do Manager.
