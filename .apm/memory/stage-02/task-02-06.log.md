# Task 2.6 - Filtros server-side na API de Transações

## Summary
O `GET /api/transactions` agora lê e aplica os filtros `date_from`, `date_to`, `category_id` e `type` no servidor, mantendo paginação 20/página, eager loading e escopo por usuário. Cobertura por feature tests dos filtros (isolados, combinados, paginação, escopo e validação).

## Details
- Criado `IndexTransactionRequest` (FormRequest) validando os query params: `date_from`/`date_to` como datas opcionais com `after_or_equal:date_from` (intervalo inclusivo, fim ≥ início); `type` como enum `entrada|saida`; `category_id` reutilizando o mesmo `Rule::exists` restrito da Task 2.1 — categoria precisa existir, não estar soft-deletada, e ser pré-definida ou do próprio usuário. Isso garante que filtrar por categoria de outro usuário resulte em 422, sem vazar dados nem existência.
- `TransactionController@index` passou a receber o `IndexTransactionRequest` e a aplicar os filtros via `when()` condicional sobre a query já escopada por `$request->user()->transactions()`, preservando `with('category')`, `latest('date')->latest('id')` e `paginate(20)`. Adicionado `withQueryString()` para que os links de paginação preservem os filtros.
- Contrato de resposta (`TransactionResource` + `meta`/`links`) e nomes dos params inalterados — encaixe imediato com a UI de histórico existente.
