# Task 2.1 - API de Transações & Categorias

## Summary
API JSON autenticada por sessão para CRUD de transações (paginado, soft delete, eager loading) e de categorias personalizadas (pré-definidas protegidas), escopada por usuário, com validação 422 em português e contrato de `amount` em centavos. 50/50 testes passando.

## Details
Despacho sequencial na pasta principal (`C:\_PROJETOS\FlowFin`), branch `feature/api-transacoes`. Construí a camada de API sobre os models da Task 1.3 e o helper `App\Support\Money` da Task 1.4.

- **Rotas:** grupo `Route::middleware('auth')->prefix('api')->name('api.')` em `routes/web.php` (sessão Breeze; CSRF aplica-se às escritas — clientes same-origin/offline enviam o token). Endpoints listados no contrato abaixo.
- **Controllers** (`app/Http/Controllers/Api/`): `TransactionController` e `CategoryController`. Escopo por usuário via consultas de relacionamento (`$request->user()->transactions()`, `->categories()`), de modo que recursos de outro usuário — ou categorias pré-definidas (que têm `user_id` nulo) — resultam em **404** no `findOrFail`, sem vazar existência.
- **FormRequests** (`app/Http/Requests/`): `StoreTransactionRequest` (+ `Update` por herança), `StoreCategoryRequest` (+ `Update`). `amount` validado como inteiro ≥ 0; `prepareForValidation` converte para centavos via `Money::toCents` apenas quando a entrada é string com `,`/`R$`. `category_id` validado com `Rule::exists` restrito a pré-definida OU do próprio usuário (e não soft-deletada). `classification` proibida quando `type=entrada` (`Rule::prohibitedIf`).
- **API Resources** (`app/Http/Resources/`): `TransactionResource`, `CategoryResource` — contrato de resposta estável; `amount` sempre em centavos; categoria embutida via `whenLoaded`.
- **Paginação/eager loading:** `index` de transações usa `with('category')` + `paginate(20)`. A listagem de categorias **não** é paginada (conjunto pequeno e limitado: 9 pré-definidas + as do usuário, usado como referência para seletores) — desvio justificado da regra geral de paginação.
- **Factories:** `CategoryFactory` (com estado `predefined()`), `TransactionFactory` (com `entrada()`/`saida()`).
- Mensagens de validação pt_BR estendidas em `lang/pt_BR/validation.php` (regras `in`, `exists`, `date`, `boolean`, `prohibited` + atributos das entidades).

## Contrato de Payload (para a UI e a camada offline)
Todos exigem sessão autenticada e header `Accept: application/json` (garante erros/404 em JSON). `amount` é **centavos inteiros** na entrada e na saída.

Transações:
- `GET /api/transactions` → `{ data: [Transaction], links, meta }` (20/página; `meta.per_page=20`).
- `POST /api/transactions` → 201 `{ data: Transaction }`. Corpo: `type` (entrada|saida, obrig.), `amount` (centavos int, obrig.), `category_id` (obrig.), `date` (aaaa-mm-dd, obrig.), `description?`, `classification?` (necessidade|desejo; proibida p/ entrada), `is_recurring?`.
- `GET /api/transactions/{id}` → `{ data: Transaction }` (404 se não for do usuário).
- `PUT /api/transactions/{id}` → `{ data }` (representação completa, mesmas regras do POST).
- `DELETE /api/transactions/{id}` → `{ message }` (soft delete).

Objeto Transaction: `{ id, type, amount(centavos), category_id, date(aaaa-mm-dd), description, classification, is_recurring, category?, created_at, updated_at }`.

Categorias:
- `GET /api/categories` → `{ data: [Category] }` (pré-definidas + do usuário; não paginado).
- `POST /api/categories` → 201 `{ data }`. Corpo: `name` (obrig.), `icon?`, `color?`. Cria sempre como personalizada do usuário.
- `PUT /api/categories/{id}` → `{ data }` (só do usuário; pré-definida → 404).
- `DELETE /api/categories/{id}` → `{ message }` (soft delete; só do usuário; pré-definida → 404).

Objeto Category: `{ id, name, icon, color, is_predefined, user_id }`.

## Important Findings
- **Contrato `amount` = centavos inteiros** na request e na response (formatação R$ é da UI). Strings R$ são toleradas na entrada (convertidas via `Money`), mas o caminho canônico é inteiro em centavos — a camada offline deve enviar centavos.
- **Erros em JSON exigem `Accept: application/json`.** As rotas vivem sob o middleware `web`/`auth` (sessão + CSRF nas escritas). Sem esse header, falhas de auth redirecionam (302) em vez de 401, e erros de validação redirecionam em vez de 422. A UI/offline deve enviar o header e o token CSRF nas escritas.
- **Listagem de categorias não paginada** (desvio justificado): é um conjunto pequeno de referência para seletores. Caso o produto permita muitas categorias personalizadas no futuro, reavaliar.
- Endpoints prontos para a UI (telas de registro/histórico) e para a futura sincronização offline do PWA — o contrato acima é a dependência dessas Tasks.
